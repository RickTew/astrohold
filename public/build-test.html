SUGGESTIONS MAPPED — WHAT TRANSLATED, WHAT DIDN'T

Both rounds of suggestions had a lot of overlap. Here's the honest accounting:

  • CanvasTexture / bitmap text → kept. The "Canvas title text" toggle draws "BUILD PHASE" into a real <canvas> with per-frame procedural flicker. No Three.js needed — the DOM has its own canvas, same API.
  • Additive blend sprites (selection ring) → kept. The "Selection pulse ring" toggle uses mix-blend-mode: screen, which is the DOM equivalent of Three.js additive blending.
  • Sprite icons (SpriteMaterial) → already done. Our tiles use <img> with pixelated rendering — visually identical, lighter.
  • Hover tint / scale animation → already done via CSS :hover. "Tile hover-pop" makes it more dramatic.
  • Procedural Layout (Group + position.lerp) → "Tile slide-in" toggle uses CSS keyframes for the same effect.
  • Shader glow / GLSL radial gradient → "Title neon boost" + "CR bloom pulse" use CSS text-shadow + filter. Same look, no shader compiler.
  • Procedural GLSL noise → SVG feTurbulence filter ("Procedural noise" toggle). This is the W3C-standard procedural-noise primitive; visually equivalent to a simplex-noise shader.
  • Pulse / Math.sin breathing → CSS @keyframes with ease-in-out infinite. Same math, declarative.
  • Scrolling starfield plane / UV-offset → "Animated starfield" toggle, CSS background-position animation.

Still inapplicable without rewriting the HUD into the Three.js scene:

  • TextGeometry / FontLoader — 3D extruded text. Different look entirely; not equivalent to canvas-text.
  • BoxGeometry / PlaneGeometry buttons — requires a 3D mesh HUD with raycaster-driven input.
  • SpotLight / RectAreaLight — Three.js lights illuminate 3D meshes, not DOM nodes.
  • UnrealBloomPass — postprocessing acts on the WebGL framebuffer, not the DOM overlay.
  • Three.Group + scene-graph hierarchy — solved at the DOM level by container <div>s; no Group is needed.
